# BHERP ERP 모듈 분리 및 메뉴 구성 계획

## 1. 결론

지금 바로 시작할 첫 번째 방향은 `메뉴/권한 기반 정리`를 0차로 먼저 하고, 실제 업무 모듈은 `인사/자격`을 1차 모듈로 삼는 것이다.

이유는 다음과 같다.

- 현재 구현된 기능이 사원, 조직, 직위, 자격증, 자격 만료 알림, 배전인력으로 이미 인사/자격 흐름에 가장 많이 모여 있다.
- ERP다운 분리는 새 기능을 많이 만드는 것보다 먼저 메뉴 체계와 권한 체계를 업무 도메인 기준으로 정리해야 효과가 난다.
- `배전인력`, `세금계산서변환`은 회사 전용 기능으로 잘 분리되어 있으므로 공통 ERP 모듈에 섞지 않고 별도 확장 메뉴로 남겨야 한다.
- 기존 메뉴가 `운영` 하나에 모두 들어가 있어, 기능이 조금만 늘어도 사용자가 화면을 찾기 어려워진다.

권장 순서:

```txt
0차: 메뉴/권한/탐색 구조 정리
1차: 인사/자격 모듈 정리
2차: 프로젝트/현장 모듈 도입
3차: 세무/회계, 매출/매입 모듈 확장
4차: 구매/재고/자산 모듈
5차: 근태/급여/전자결재/리포트 고도화
```

## 2. 현재 구조 진단

### 2.1 이미 좋은 점

- 백엔드는 NestJS feature module 단위로 나뉘어 있다.
- 프론트도 `features/operation/<feature>` 패턴이 잡혀 있다.
- 메뉴 권한은 `MenuPermissionGuard`, `@MenuCode`, `@Permission`으로 이미 ERP식 권한 제어 기반이 있다.
- 메뉴 DB에는 `parentId`, `icon`, `sortOrder`가 있어 트리형 메뉴로 확장할 수 있다.
- MDI 탭과 메뉴 검색 기능이 이미 설계 및 일부 구현되어 있어, 메뉴가 많아져도 업무 UX를 유지할 수 있다.
- 회사 전용 기능은 `company-features`로 격리되어 있어 공통 ERP와 분리하기 좋다.

### 2.2 지금 막히는 지점

- 프론트 메뉴는 `운영` 단일 그룹 안의 평면 목록이다.
- 백엔드 메뉴 seed도 `OPERATION` 단일 그룹 기준이다.
- 메뉴 코드가 모두 `OP_*`라서 앞으로 HR, 회계, 프로젝트, 시스템 메뉴가 늘어나면 의미가 흐려진다.
- 프론트 라우트도 대부분 `/operation/...` 아래에 있어 업무 모듈 경계가 URL에 드러나지 않는다.
- 공통코드, 자동채번, 메뉴관리 같은 ERP 공통 관리 화면이 아직 메뉴로 드러나지 않는다.
- 사원은 `departmentName`, `positionName` 문자열을 들고 있어 조직/직위 기준정보와의 정합성이 약하다.

## 3. 모듈 분리 원칙

ERP 모듈은 CRUD 테이블 기준이 아니라 업무 책임 기준으로 나눈다.

권장 원칙:

- 기준정보는 여러 업무가 공유하는 마스터 데이터로 둔다.
- 인사, 프로젝트, 회계, 구매, 재고처럼 사용자의 업무 흐름이 다른 영역은 별도 모듈로 둔다.
- 회사 전용 자동화는 공통 ERP에 섞지 않고 `회사전용/확장업무`로 둔다.
- 메뉴는 최대 3단계까지만 허용한다. 4단계 이상은 검색으로 찾게 한다.
- 메뉴가 많아지는 것은 드롭다운 하나로 해결하지 말고 `사이드바 아코디언 + 상단 검색 + 즐겨찾기/최근 메뉴 + MDI 탭`을 같이 사용한다.
- 기존 `menuCode`는 권한과 연결되어 있으므로 바로 대규모 변경하지 않는다. 0차에서는 메뉴 그룹과 표시 구조부터 정리하고, 코드명 변경은 별도 마이그레이션으로 미룬다.

## 6. 메뉴 UX 방향

### 6.1 데스크톱

권장 구조:

```txt
상단 헤더
  ERP명 / 메뉴 검색 / 알림 / 사용자

좌측 사이드바
  Dashboard
  기준정보       접힘/펼침
  인사/자격      접힘/펼침
  프로젝트/현장   접힘/펼침
  세무/회계      접힘/펼침
  시스템         접힘/펼침

본문
  MDI 탭
  활성 업무 화면
```

상단 드롭다운은 보조 진입점으로 둔다.

권장:

- 좌측은 업무 중 계속 보이는 기본 네비게이션
- 상단 검색은 메뉴가 많아질 때 가장 빠른 진입점
- 상단 `업무 메뉴` 드롭다운은 모듈 전체를 빠르게 훑는 용도
- MDI 탭은 열린 업무를 전환하는 용도
- 즐겨찾기/최근 메뉴는 2차 이후 추가

### 6.2 모바일

권장 구조:

```txt
상단 헤더
  햄버거 / ERP명 / 검색 아이콘 / 알림

메뉴 Drawer
  모듈 아코디언
  메뉴 검색
```

모바일에서는 메뉴 트리가 길어지면 스크롤이 커지므로 검색을 더 앞에 둔다.

### 6.3 드롭다운만으로 해결하지 않는 이유

ERP는 메뉴가 30개를 넘어가면 단순 드롭다운이 금방 불편해진다.

따라서 다음 조합이 좋다.

- 반복 업무: MDI 탭
- 새 업무 진입: 좌측 아코디언
- 메뉴가 기억나지 않을 때: 상단 메뉴 검색
- 자주 쓰는 메뉴: 즐겨찾기
- 방금 쓴 메뉴: 최근 메뉴

## 7. 데이터/코드 구조 변경 방향

### 7.1 메뉴 타입 확장

프론트 `MenuItem`은 2단계 이상 메뉴를 표현할 수 있게 확장한다.

예상 타입:

```ts
export type MenuNode = {
  menuCode: string;
  title: string;
  path?: string;
  icon?: string;
  moduleCode: string;
  children?: MenuNode[];
  featureFlag?: string;
  mdi?: {
    closable?: boolean;
    pinned?: boolean;
    reuse?: "by-menu-code" | "by-path";
  };
};
```

### 7.2 백엔드 메뉴 엔티티

현재 `Menu`에는 `parentId`가 있지만, 상위 그룹 메뉴도 제대로 표현하려면 다음 보강이 좋다.

후보:

- `moduleCode`: `MASTER_DATA`, `HR`, `PROJECT`, `ACCOUNTING`, `SYSTEM`
- `menuType`: `GROUP`, `MENU`
- `path`: 그룹 메뉴에서는 nullable 허용
- `depth`: 선택 사항. `parentId`로 계산해도 된다.
- `isVisible`: 메뉴에는 숨기되 권한 체크에는 쓰는 경우 대비

주의:

- 기존 DB에는 `path`가 필수라 그룹 메뉴를 넣기 애매하다.
- 0차에서는 프론트 config만 계층화하고, 백엔드 DB 메뉴 트리는 1차에서 정리해도 된다.

### 7.3 라우트 정책

처음부터 모든 라우트를 바꾸면 리스크가 크다.

권장 단계:

1. 0차: 기존 `/operation/...` 라우트 유지, 메뉴 표시 그룹만 재구성
2. 1차: 신규 화면부터 `/hr/...`, `/master-data/...`, `/system/...` 형식 사용
3. 2차: 기존 화면을 새 라우트로 옮기고 기존 경로는 redirect 제공
4. 3차: 메뉴 코드와 라우트명을 도메인 기준으로 정리

예시:

```txt
/operation/employees                  유지
/hr/employees                         신규 표준 경로
/operation/employees -> /hr/employees redirect
```

### 7.4 메뉴 코드 정책

기존 권한 데이터와 연결되어 있으므로 `OP_*` 코드를 당장 바꾸지 않는다.

신규 메뉴부터 다음 규칙을 적용한다.

```txt
MD_*   기준정보
HR_*   인사/자격
PJ_*   프로젝트/현장
```

## 8. 0차 계획: 메뉴/권한/탐색 구조 정리

목표:

- 현재 메뉴를 업무 모듈 기준으로 재배치한다.
- 메뉴가 늘어나도 버틸 수 있는 UI 구조를 만든다.
- MDI 탭과 메뉴 검색이 새 메뉴 구조를 그대로 사용할 수 있게 한다.

작업:

1. `frontend/src/types/menu.ts` 계층 메뉴 타입 확장
2. `frontend/src/config/menus.ts`를 모듈/섹션/메뉴 구조로 재정의
3. `MenuGroup`을 아코디언/트리 메뉴로 변경
4. 사이드바에서 2단계 메뉴 표시 지원
5. 메뉴 검색이 `children`까지 평탄화해 검색하도록 변경
6. 검색 결과에 `기준정보 > 조직/인사 기준 > 부서등록` 같은 breadcrumb 표시
7. MDI `openTab(menu)`은 leaf menu만 열도록 보정
8. feature flag 메뉴도 계층 구조 안에서 동작하게 처리
9. 백엔드 `menus.seed.ts`는 당장 최소 변경 또는 1차로 이월
10. 빌드 검증

0차 완료 기준:

- 같은 메뉴를 다시 열어도 MDI 탭이 중복 생성되지 않는다.
- 상단 검색으로 모든 leaf 메뉴를 찾을 수 있다.
- 배전인력, 세금계산서변환 feature flag가 유지된다.
- 모바일 Drawer에서 메뉴가 깨지지 않는다.

## 16. 리스크와 주의사항

- 메뉴 코드 변경은 권한 데이터와 연결되어 있으므로 한 번에 바꾸면 위험하다.
- 백엔드 메뉴 DB와 프론트 메뉴 config가 계속 따로 가면 나중에 불일치가 생긴다. 결국 DB를 단일 출처로 삼을지 결정해야 한다.
- 상위 메뉴에 path가 없을 수 있으므로 기존 `path` 필수 구조를 그대로 쓰면 트리 메뉴 구현이 어색해진다.
- 메뉴가 많아질수록 검색, 즐겨찾기, 최근 메뉴가 없으면 사용성이 떨어진다.
- 회사 전용 기능을 공통 모듈로 빨리 섞으면 다른 회사용 포킹이 어려워진다.
- 사원 개인정보와 세금계산서 거래처 정보는 로그/화면/엑셀 다운로드에서 계속 주의해야 한다.

## 17. 결정 필요 사항

다음은 실제 구현 전에 결정하면 좋다.

1. 0차에서 백엔드 메뉴 DB까지 계층화할지, 프론트 메뉴만 먼저 바꿀지
   - 권장: 프론트 메뉴 먼저 변경, 백엔드 DB 계층화는 1차에서 진행
2. 기존 `/operation/...` 라우트를 바로 바꿀지
   - 권장: 당장은 유지하고, 새 화면부터 도메인 라우트 사용
3. `배전인력`과 `세금계산서변환`을 어느 메뉴에 둘지
   - 권장: 화면상으로는 관련 업무 모듈 아래에 두되 코드와 폴더는 회사전용 유지
4. 공통코드/자동채번/메뉴관리 화면을 1차에 포함할지
   - 권장: 공통코드와 자동채번은 1차, 메뉴관리는 1차 후반 또는 2차
5. 사원 부서/직위를 FK로 바꿀지
   - 권장: 인사/자격 1차에서 설계 후 점진 전환

## 18. 최종 권장안

최종적으로는 다음 순서가 가장 현실적이다.

1. 메뉴 구조부터 바꾼다.
2. 현재 구현된 화면을 ERP 모듈 기준으로 재배치한다.
3. 첫 업무 모듈은 인사/자격으로 잡는다.
4. 그 다음 프로젝트/현장을 만든다.
5. 세무/회계는 성원전기 전용 변환기와 공통 회계 모듈을 분리해서 확장한다.
6. 구매/재고/자산은 프로젝트와 거래처/품목 기준정보가 잡힌 뒤 시작한다.

이 방식이면 한 번에 거대한 ERP를 만들려다 멈추지 않고, 지금 있는 기능을 바로 ERP답게 보이게 만들면서 차수별로 확장할 수 있다.
